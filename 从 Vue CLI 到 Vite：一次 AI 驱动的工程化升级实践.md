# 从 Vue CLI 到 Vite：一次 AI 驱动的工程化升级实践

## 背景

由于历史原因，项目最初基于 Vue2 + Vue CLI
搭建。在当时，这是一个稳定且成熟的技术选型，广泛应用于各类前端web应用中。

但随着前端工程体系的发展，这一组合逐渐暴露出一些问题：

- 构建速度慢
- 热更新体验一般
- 配置黑盒化严重
- 对现代前端工具链（如 ES
  Modules、按需加载、原生构建能力等）的支持不够理想

与此同时，以 Vite 为代表的新一代构建工具逐渐成为主流。它基于浏览器原生
ESM，借助 Rollup 构建生产环境，带来了极致的开发体验。

但对于一个已有一定规模的 Vue2 项目来说，从 Vue CLI 迁移到
Vite5，并不是简单的工具替换，而是一场涉及构建体系、依赖管理、插件生态甚至代码结构的系统性升级。

这一次，我尝试了以 AI 为主导，完成整个升级过程。 不是传统意义上的"AI 辅助"，而是接近于一种"AI
驱动开发（AI vibe coding）"的实践。

---

## 提示词设计：让 AI 成为工程参与者

升级的第一步不是改代码，而是设计提示词（Prompt Engineering）。

相比零散提问，这次采用的是**任务导向型提示词**，核心包括：

- 项目现状（Vue2 + Vue CLI）
- 目标（迁移到 Vite5）
- 约束（不升级 Vue3，优先保证稳定）
- 输出期望（分步骤执行、可验证）

核心关键在于：

> 如何让 AI 理解上下文，并持续保持一致性

---

## 前期规划：用 Superpowers Skill 构建迁移蓝图

在真正开始迁移之前，我没有让 AI 直接写代码，而是先做了一件更重要的事：

> 让 AI 先"想清楚"，再"动手"

### 3.1 为什么选择 Superpowers Skill

- 输出分阶段执行计划
- 自动补全风险点
- 提供工程化迁移路径
- 更适合复杂任务

本质上，它更像：

> 一个具备架构能力的技术负责人

---

### 3.2 迁移规划内容

#### （1）构建体系迁移

- Vue CLI（Webpack） → Vite5
- vue.config.js → vite.config.ts
- alias、环境变量、静态资源处理

#### （2）依赖与插件兼容

- 标记 webpack-only 插件
- 提供 Vite 替代方案
- 检查 polyfill

#### （3）代码层适配

- CommonJS → ESModule
- 动态 import
- process / global 替换

#### （4）风险与回滚

- 不可逆步骤
- 分阶段验证
- Git 回滚

---

### 3.3 从“建议”到“可执行方案”

Superpowers Skill 的核心价值在于：

> 输出的是“步骤清单”，而不是“建议”

例如：

```text
Step 1: 初始化 Vite 项目
Step 2: 迁移入口文件
Step 3: 替换路由配置
Step 4: 迁移业务模块
Step 5: 修复构建问题
```

这让迁移过程从：

> 探索式试错 → 有路线图的工程推进

## 执行阶段：用 Context7 构建可信代码

核心问题：

> AI 生成的代码是否可信？

在 Vue CLI → Vite5 的迁移中，这个问题被放大：

- 构建工具切换（Webpack → Vite）
- API 存在版本差异
- 插件生态变化
- 历史经验失效

---

### 为什么必须使用 Context7

AI 的问题不是不会写代码，而是：

> 不知道当前上下文中什么是“正确的代码”

常见问题：

- 使用废弃 API
- 混用版本配置
- 引用旧方案

这些问题的特点是：

> “看起来对，但实际上不对”

### Context7 的核心作用

#### 提供“唯一可信 API 来源”

确保：

- defineConfig
- 插件注册方式
- import.meta.env
- 构建配置结构

全部符合官方语义

#### 识别 breaking changes

例如：

- Webpack loader → Vite plugin
- process.env → import.meta.env
- 静态资源处理变化

避免：

> 用旧世界的认知进入新体系

#### 降低 AI 幻觉

约束 AI：

- 不允许虚构 API
- 不允许拼接错误配置
- 必须可在官方文档中验证

---

### 执行闭环

形成完整流程：

```text
需求 → Context7 → 代码生成 → 运行 → 验证 → 修复
```

职责划分：

- Context7：保证信息正确
- AI：提升实现效率
- Playwright：验证结果

---

## 线上问题处理：Sentry + Claude Code Plugin

升级上线后，真正挑战才开始。

传统流程：

> 错误捕获 → 定位 → 修复 → 发布

现在变成：

> 错误捕获 → AI 分析 → 修复建议 → 辅助或直接修复 -> 发布

### AI 接管分析（Claude Code Plugin）

流程：

- 拉取错误
- 解析 stack trace
- 定位代码
- 结合项目上下文推理

关键变化：

> AI 在“完整上下文”中理解问题,给出分析结果，给出分析建议，辅助或直接修复bug

### 风险与边界

仍需控制：

- 核心逻辑需人工 review
- 复杂业务问题仍需人工决策

> AI 是“高效值班工程师”，不是最终决策者

## 总结与收获

这次升级，本质上不是工具迁移，而是：

> 开发范式的转变

从：

> 人主导 → AI 辅助

到：

> AI 主导 → 人做决策

---

## 技术影响力：从实践到可复制能力

这次实践的价值，不在于“完成了一次迁移”，而在于：

> 沉淀出一套可复用的 AI 工程模式

该模式可以推广到：

- 框架升级（Vue2 → Vue3 / React 迁移）
- 构建工具迁移（Webpack → Vite）
- 大规模代码重构
- 技术债治理

---

## 结语

当 AI 开始参与：

- 工程决策
- 代码实现
- 自动验证
- 问题分析修复

前端开发已经不再只是“写代码”，而是：

> 构建一个人机协同的工程系统

Vue CLI 到 Vite 的迁移只是一个切入口。

真正的变化，是我们正在重新定义：

> 我们正在从"写代码"，走向"构建智能工程系统"。
